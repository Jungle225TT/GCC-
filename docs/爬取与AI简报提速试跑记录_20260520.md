> 试跑时间：2026-05-20 20:33-21:05  
> 目的：按当前优化方案连续运行 3 次，记录实际耗时、文章数量和稳定性。

---

## 一、试跑命令

三次试跑均使用同一组参数：

```bash
python3 gcc_thinktank_scraper_v2.py \
  --dry-run-keywords \
  --no-dedup \
  --playwright \
  --ai \
  --days 30 \
  --max-per-tank 50 \
  --summary-workers 2 \
  --output-dir /private/tmp/gcc_speed_trials/run_N
```

说明：

- 使用 `--no-dedup`，避免第 2、3 次因为 SQLite 去重而没有文章可处理。
- 使用 `--summary-workers 2`，验证 AI 简报分批并发生成效果。
- 每次输出到独立目录，便于对比日志和产物。

## 三、关键观察

### 1. 提速目标达成

方案中保守预估完整流程约 11-15 分钟，本次三轮实际平均约 10.6 分钟，略好于预期。

对比优化前约 35 分钟的“先 dry-run 再正式 AI 跑”流程，当前完整流程大约缩短到原来的三分之一。

### 2. Carnegie 优化稳定生效

三轮 Carnegie 指标完全一致：

| 指标 | 数值 |
|------|------|
| Carnegie 候选文章 | 135 |
| 实际访问详情页 | 3 |
| 跳过后续必过滤文章 | 132 |

这说明“先判断时间窗口，再做 metadata 验证”的优化稳定减少了大量无效详情页访问。

### 3. AI 简报并发稳定

三轮 AI 简报均为 7 批生成，未出现失败批次。并发 2 可以正常运行，并且最终会按原始批次顺序拼回正文。

第 3 次 AI 简报生成耗时较长，主要是 DeepSeek 响应速度波动；这属于外部服务耗时，不是本地筛选逻辑问题。

### 4. 简报新增附录已生效

三轮生成的 AI 简报均包含：

- `关键词硬过滤复核清单`：来自 `filtered_out.csv`，本次每轮 23 条。
- `合规未自动抓取但建议人工关注的来源`：来自 `blocked_sources.csv`，本次每轮 3 个来源。

这满足“被硬过滤内容也要在简报中呈现”和“法律/政策无法自动抓取但值得人工查看的官网链接要整理出来”的要求。

## 四、波动来源

本次主要波动来自两类外部因素：

1. 网站响应波动  
   GRC 多次 timeout；MEI、CSIS 多次 403；Chatham House 需要 requests 失败后走 Playwright 兜底。

2. AI 响应波动  
   第 3 次 AI 简报生成耗时 270.8 秒，高于第 1、2 次的 202.1 秒和 195.3 秒。

文章数量没有因此下降，说明当前流程对这些波动有一定韧性。

## 六、建议

1. `--summary-workers 2` 可以作为当前默认参数保留。
2. 如果后续 DeepSeek 偶发限流，再回退为 `--summary-workers 1`。
3. 下一步优化重点不是 AI，而是少数慢站点：
   - GRC timeout 策略可继续收紧；
   - 对长期 403 的站点可考虑是否改为人工入口或降低抓取频率；
   - RSS 可用的站点继续优先 RSS。
4. 若做每周自动运行，建议继续记录这四项核心指标：总耗时、最终文章数、强相关数、AI 批次失败数。
