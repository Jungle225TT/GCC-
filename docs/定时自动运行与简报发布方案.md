# GCC智库抓取系统定时自动运行与简报发布方案

> 编写时间：2026-05-18  
> 目标：摆脱 Google Colab 手动运行和手动上传飞书，实现每日定时抓取、AI 简报生成、自动发布到员工可访问位置。

---

## 1. 当前问题

目前流程是：

1. 在 Google Colab 手动打开笔记本。
2. 挂载 Google Drive、安装依赖、运行抓取脚本。
3. 生成 Markdown、JSON、AI 简报 PDF。
4. 人工上传到飞书。

主要痛点：

- Colab 每次冷启动都要重新安装依赖和 Playwright Chromium，耗时明显。
- Colab 免费资源不稳定，Playwright 串行跑多个网站时更慢。
- 人工上传飞书容易漏传、传错版本，无法形成稳定日报链路。
- 本地 SQLite 去重和运行日志不容易长期沉淀。
- 目前飞书同步主要同步文章列表，还没有自动发布 AI 简报正文/PDF 链接。

---

## 2. 目标能力

最低可用版本应满足：

- 每天固定时间自动运行。
- 自动抓取近 1-3 天文章，避免每日重复处理 30 天窗口。
- 自动生成 `gcc_research_*.json`、`gcc_research_*.md`、`gcc_summary_*.md`、`gcc_summary_*.pdf`。
- 自动同步文章列表到飞书多维表格。
- 自动把 AI 简报上传到一个公司员工可访问的位置，并把链接写回飞书。
- 失败时有日志、有通知、有手动重跑入口。
- API Key、飞书凭证不写入代码仓库。

理想版本进一步满足：

- 保留持久化去重数据库和历史运行日志。
- 自动归档每日简报。
- 支持日报、周报、专项国家调试三种运行模式。
- 高风险来源默认不抓取，人工覆盖需要显式参数。
- 出错后能重试，长期无人值守。

## 4. 方案 A：继续 Google Colab

### 做法

保留现在的 `notebooks/GCC_Scraper_Colab.ipynb`，每天人工或由负责人手动运行，再上传飞书。

### 优点

- 不需要新基础设施。
- 研究员可见、可点、可调试。
- 适合临时测试关键词、检查某个来源。

### 缺点

- 无法真正无人值守。
- 每次运行环境都是临时的，依赖安装和 Chromium 安装慢。
- 长任务容易受 Colab 空闲断开、资源限制影响。
- 运行日志和去重状态不稳定。
- 仍然需要人工上传简报。

### 结论

只保留为调试入口，不建议作为每日生产任务。

---

## 5. 方案 B：GitHub Actions + 飞书自动发布

### 做法

在仓库新增 `.github/workflows/gcc_daily.yml`：

- 每天北京时间早上运行一次。
- 安装 Python 依赖和 Playwright Chromium。
- 执行：

```bash
python gcc_thinktank_scraper_v2.py --playwright --ai --days 2
python feishu_sync.py
python publish_summary.py
```

其中 `publish_summary.py` 是建议新增脚本，用于：

1. 找到最新 `output/gcc_summary_*.md` 和 `output/gcc_summary_*.pdf`。
2. 上传到飞书云文档、飞书文件夹、知识库或多维表格附件字段。
3. 生成员工可访问链接。
4. 将链接写入飞书多维表格的“每日简报索引”表。

### 优点

- 不需要买服务器，部署最快。
- GitHub Secrets 可以安全保存 `DEEPSEEK_API_KEY` 和飞书凭证。
- 支持 `workflow_dispatch` 手动重跑。
- 输出文件可作为 GitHub Actions artifact 留档。
- 适合当前项目规模：每天一次、几十到几百篇候选文章。

### 缺点

- GitHub runner 是临时环境，每次仍要安装依赖和 Chromium。
- SQLite 去重数据库不会自然持久化，需要用飞书已有 URL 或 artifact/cache 辅助。
- 定时触发不是严格实时，GitHub 官方说明 schedule 在高负载时可能延迟，甚至极端情况下丢弃排队任务。
- 私有仓库使用 GitHub-hosted runner 有免费分钟和 artifact storage 配额，超出后可能计费。
- 出口 IP 不固定，个别站点可能偶发 403。

### 成本判断

按当前任务每天一次估算：

- 每次运行约 10-30 分钟，取决于 Playwright、AI 简报和网络。
- 每月约 300-900 runner 分钟。
- GitHub 官方文档显示 GitHub Free 对私有仓库有每月 2,000 分钟和 500 MB artifact storage 的额度；实际额度和计费以账号计划为准。
- DeepSeek 调用成本目前很低，通常远低于云服务器成本。
- 飞书 API 本身通常不是主要成本，主要成本是企业应用配置和权限维护。

### 适配建议

- cron 不要设在整点，例如北京时间 08:17，对应 UTC `17 0 * * *`，降低 GitHub 高峰延迟概率。
- `--days` 建议日报用 `2` 或 `3`，不要每天全量跑 `30` 天。
- 用飞书表格已有 URL 作为最终去重依据，避免 GitHub runner 临时环境造成重复推送。
- GitHub artifact 只保留 7-14 天，长期留档放飞书或对象存储。

### 结论

这是最推荐的第一阶段方案。投入低、上线快、足够解决“Colab 慢 + 手动上传”的问题。

## 6. 方案 C：GitHub Actions + GitHub Pages/Artifacts

### 做法

GitHub Actions 定时生成简报后：

- Markdown/PDF 作为 artifact 保存；
- 或发布到 GitHub Pages；
- 员工通过 GitHub 链接访问。

### 优点

- 技术实现简单。
- 不依赖飞书文件上传 API。
- 适合研发团队查看日志和产物。

### 缺点

- 公司员工未必都有 GitHub 账号或仓库权限。
- 私有仓库下 Pages/Artifacts 的访问管理不如飞书自然。
- 简报阅读场景和公司日常协作工具脱节。

### 结论

可作为备份留档，不建议作为员工主入口。

## 7. 方案 D：轻量云服务器 + cron + 飞书

### 做法

购买一台低配 Linux 云服务器，安装依赖后长期运行：

```bash
cd /opt/GccScraper
python -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
playwright install chromium
```

crontab：

```cron
17 8 * * * cd /opt/GccScraper && . .venv/bin/activate && python gcc_thinktank_scraper_v2.py --playwright --ai --days 2 && python feishu_sync.py && python publish_summary.py >> logs/daily.log 2>&1
```

### 优点

- 环境常驻，依赖和 Chromium 不需要每天重装。
- SQLite 去重数据库、日志、历史产物可以长期保存。
- 运行速度通常比 Colab 和每次冷启动的 GitHub Actions 更稳定。
- 可加 systemd timer、失败重试、磁盘监控、企业微信/飞书告警。
- 后续如果加全文抓取、更多来源、向量检索，也更容易扩展。

### 缺点

- 需要购买和维护服务器。
- 需要处理系统安全、密钥、备份、磁盘清理。
- 负责人需要具备基本 Linux 运维能力。

### 成本判断

如果只跑当前脚本，低配 1-2 vCPU、2-4 GB 内存通常够用；Playwright 会占用更多内存，建议不要低于 2 GB。国内云厂商或海外 VPS 的具体价格波动较大，应以采购当天报价为准。长期看，若每天都跑、还要保留日志和数据库，云服务器的稳定性和维护价值通常高于 Colab。

### 结论

这是第二阶段推荐方案。等 GitHub Actions 跑通流程后，再迁移到云服务器，会更稳。

## 8. 方案 E：公司内网机器或 Mac mini

### 做法

找一台长期联网、不关机的公司电脑或 Mac mini，配置 Python 环境，用 cron/任务计划程序定时运行。

### 优点

- 不需要额外云服务器费用。
- 数据留在公司设备上。
- 如果网络访问海外智库较稳定，运行速度不错。

### 缺点

- 依赖设备不断电、不休眠、不断网。
- 系统升级、人员使用、网络策略变化都可能影响任务。
- 日志、告警、远程维护能力弱于云服务器。

### 结论

如果研究院已有稳定常开设备，可以作为低成本生产方案；否则不如云服务器省心。

---

## 9. 方案 F：Serverless/云函数

### 做法

用云函数或容器任务定时触发抓取。

### 优点

- 理论上按量付费。
- 无需维护完整服务器。
- 定时触发和日志能力较成熟。

### 缺点

- Playwright/Chromium 打包和运行复杂。
- 函数执行时长、内存、临时磁盘可能受限。
- 依赖安装和浏览器环境问题会吃掉大量调试时间。
- 当前脚本是单文件批处理，更适合 runner 或服务器，不适合拆成短函数。

### 结论

除非后续改造成容器化任务，否则不建议作为主方案。

---

## 10. 方案 G：Airflow/Prefect 等调度平台

### 做法

将抓取、AI 简报、飞书发布拆成多个 DAG/task，由专业调度系统管理。

### 优点

- 任务依赖、重试、监控、历史记录完善。
- 适合未来同时管理多个爬虫、数据清洗和报告任务。

### 缺点

- 当前项目规模下明显过重。
- 部署和维护成本高。
- 对研究团队来说学习成本较大。

### 结论

暂不需要。等项目扩展到多个稳定数据管线后再考虑。

## 11. 推荐路线

### 第一阶段：GitHub Actions 快速上线

时间：1-2 天。

目标：

- 每天自动跑。
- 自动同步文章列表到飞书。
- 自动上传 AI 简报，并在飞书写入链接。
- 保留 Colab 作为调试入口。

需要新增：

- `.github/workflows/gcc_daily.yml`
- `publish_summary.py`
- 飞书多维表格“每日简报索引”表，字段建议：
  - 日期
  - 运行时间
  - 文章总数
  - 强相关篇数
  - 中等相关篇数
  - AI 简报 Markdown 链接
  - AI 简报 PDF 链接
  - 运行状态
  - 错误信息

建议运行参数：

```bash
python gcc_thinktank_scraper_v2.py --playwright --ai --days 2
```

保守理由：

- 日报不需要每天重扫 30 天。
- 当前已有飞书 URL 去重，可防止重复写入文章。
- 高风险来源已由 `compliance_rules.yaml` 默认跳过。

### 第二阶段：补齐发布和监控

时间：3-5 天。

目标：

- 简报上传失败时，不影响文章列表同步。
- GitHub Actions artifact 保存 7-14 天。
- 飞书表格记录每次运行状态。
- 飞书机器人或群通知每日简报链接。
- 增加“无新文章”时的空日报逻辑。

### 第三阶段：迁移到轻量云服务器

时间：1-2 周后评估。

迁移条件：

- GitHub Actions 每天运行时间经常超过 30 分钟。
- GitHub 免费分钟接近上限。
- 需要长期保留 SQLite 去重数据库和日志。
- 需要更稳定地处理 Playwright 站点。
- 需要未来扩展全文抓取、向量检索或多份专题报告。

## 12. 最终结论

综合速度、成本、维护和员工访问便利性：

1. **立即采用 GitHub Actions + 飞书自动发布。**  
   这是当前性价比最高的方案：无需买服务器，能快速替代 Colab 手动运行和人工上传。

2. **员工查看入口继续放在飞书。**  
   飞书是公司已有协作环境，文章列表、每日简报索引、PDF 链接都放飞书，员工无需学习 GitHub。

3. **Colab 降级为调试工具。**  
   以后只用于关键词调参、单国测试、临时复现，不再承担每日生产任务。

4. **运行稳定后迁移到轻量云服务器。**  
   如果每日任务变重，或需要更强日志、去重和扩展能力，云服务器会比 GitHub Actions 更稳。
