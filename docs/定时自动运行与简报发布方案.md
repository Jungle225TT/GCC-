# GCC智库抓取系统每周自动运行与简报发布方案

> 更新时间：2026-05-19
> 用途：供小组内部讨论，并向项目负责人说明后续自动化运行方案。

---

## 一、为什么改成每周运行

目前这套系统已经能完成智库文章抓取、筛选、AI 简报生成和 PDF 导出，但运行方式还比较依赖人工：

1. 需要手动打开 Google Colab。
2. 每次都要等待环境和浏览器依赖安装。
3. 简报生成后，还需要人工上传到飞书。
4. 如果中途失败，需要人工查日志、重跑。

从实际使用看，智库文章并不是每天都有大量高价值更新。每天跑一次，成本和耗时都会增加，但对阅读效果的提升有限。相比之下，每周固定生成一份简报，更符合研究团队和公司同事的阅读节奏，也更便于形成稳定归档。

所以建议把正式生产任务设为“每周运行一次”，Colab 只保留为临时调试工具。

---

## 二、每周运行的基本目标

建议每周一上午自动生成上一周期的研究简报。核心目标如下：

| 项目 | 建议设置 |
|------|----------|
| 运行频率 | 每周 1 次 |
| 建议时间 | 北京时间每周一 08:17 |
| 抓取范围 | 近 14 天文章 |
| 去重窗口 | 21 天 |
| 输出位置 | 飞书知识库或飞书文件夹 |
| 员工查看方式 | 飞书链接 + 每周简报索引表 |
| 保留文件 | PDF、Markdown、JSON、质检文件 |

这里建议用 14 天作为抓取窗口，是为了给网站更新延迟留出余量。如果某些文章发布时间较晚，或者某周任务失败，下次运行仍然有机会覆盖到。21 天去重则可以避免重复推送已经处理过的文章。

---

## 三、每周会生成哪些文件

每次运行后，建议保留以下文件：

| 文件 | 作用 |
|------|------|
| `gcc_summary_*.pdf` | 面向同事阅读的正式 AI 简报 |
| `gcc_summary_*.md` | 简报 Markdown 原文，方便后续修改或复制 |
| `gcc_research_*.md` | 文章列表，包括强相关和中等相关分组 |
| `gcc_research_*.json` | 结构化数据，便于后续检索或同步 |
| `keyword_dryrun.json` | 进入结果池文章的评分记录 |
| `filtered_out.csv` | 被过滤掉的文章，方便检查是否误删 |
| `blocked_sources.csv` | 因合规原因主动避开的智库来源 |

其中，`filtered_out.csv` 和 `blocked_sources.csv` 主要给我们内部质检使用。正式给员工看的，还是 PDF 简报和飞书索引。

## 四、几个可选方案

| 方案 | 做法 | 优点 | 问题 | 判断 |
|------|------|------|------|------|
| 继续用 Colab | 每周人工打开 notebook 跑一次 | 不需要新环境 | 仍然慢，也还是人工上传 | 只适合调试 |
| GitHub Actions + 飞书 | GitHub 每周自动运行，结果上传飞书 | 上线快、成本低、可手动重跑 | 每次运行仍是临时环境 | 建议第一阶段采用 |
| 轻量云服务器 + 飞书 | 买一台小服务器，定时运行 | 环境稳定、日志和数据库好保存 | 需要运维和费用 | 第二阶段再考虑 |
| 公司常开电脑 | 用办公室固定设备定时跑 | 不额外买服务器 | 依赖设备不断电、不休眠 | 有稳定设备时可选 |
| 云函数/调度平台 | 用 Serverless、Airflow 等 | 长期可扩展 | 对当前项目过重 | 暂不建议 |

综合看，目前最合适的是先用 **GitHub Actions + 飞书自动发布**。它可以较快替代 Colab 手动流程，也不需要马上采购服务器。

## 五、推荐落地方案

### 1. 运行方式

在 GitHub 仓库里配置一个每周任务，例如 `.github/workflows/gcc_weekly.yml`。

GitHub 的定时任务使用 UTC 时间。若希望北京时间每周一 08:17 运行，对应配置为：

```yaml
on:
  schedule:
    - cron: "17 0 * * 1"
  workflow_dispatch:
```

`workflow_dispatch` 是手动重跑入口。如果某周需要补跑，不必再打开 Colab，可以直接在 GitHub 页面点运行。

### 2. 运行命令

建议生产命令如下：

```bash
python gcc_thinktank_scraper_v2.py \
  --dry-run-keywords \
  --playwright \
  --ai \
  --days 14 \
  --dedup-days 21 \
  --max-per-tank 50 \
  --summary-workers 2 \
  --output-dir ./output
```

说明：

- `--days 14`：每次看近 14 天文章。
- `--dedup-days 21`：近 21 天处理过的文章不重复推送。
- `--dry-run-keywords`：同时生成质检文件，方便回看筛选结果。
- `--summary-workers 2`：AI 简报分批并发生成，速度会比单批串行更快。
- 手动复盘同一批文章时，才加 `--no-dedup`。

### 3. 发布方式

建议新增一个发布脚本 `publish_summary.py`，负责把本次生成的文件上传到飞书。

发布流程可以先做简单版本：

1. 找到最新生成的 PDF、Markdown 和质检文件。
2. 上传到指定飞书文件夹或知识库。
3. 生成员工可访问链接。
4. 把链接写入飞书多维表格。
5. 可选：在飞书群里发一条简短通知。

---

## 六、飞书中如何呈现

建议在飞书里建一个“GCC智库每周简报”入口，形式可以是文件夹或知识库页面。

同时建一张“每周简报索引”多维表格，字段建议如下：

| 字段 | 内容 |
|------|------|
| 周期 | 例如 `2026-W21` |
| 运行时间 | 自动记录 |
| 抓取窗口 | 近 14 天 |
| 收录文章数 | 本次进入简报的文章数 |
| 强相关篇数 | 简报中置顶推荐的篇数 |
| 中等相关篇数 | 其余背景参考文章篇数 |
| AI 简报 PDF | 员工主要阅读入口 |
| AI 简报 Markdown | 便于研究团队二次编辑 |
| 质检文件 | `filtered_out.csv` 等文件链接 |
| 默认跳过来源 | `blocked_sources.csv` 链接 |
| 运行状态 | 成功、失败、手动重跑 |
| 备注 | 记录异常情况 |

这样做的好处是，大家只需要打开飞书链接阅读简报；我们内部也能保留完整运行记录，后续排查问题会方便很多。

---

## 七、质检和合规处理

每周运行后，需要保留三类辅助文件：

1. `keyword_dryrun.json`：看文章为什么被保留。
2. `filtered_out.csv`：看哪些文章被过滤，是否存在误删。
3. `blocked_sources.csv`：看哪些来源因服务条款或自动化访问限制被主动避开。

目前系统会默认跳过 Brookings、Atlantic Council、IISS 等高风险来源，并在简报末尾加说明：这些来源不是技术上没有抓到，而是出于合规考虑未做自动抓取。若同事对这些机构内容感兴趣，可以通过官网链接自行阅读。

这部分建议继续保守处理。只要对方条款明确反对自动化抓取，就不抓标题、链接、摘要或正文。

## 八、后续分阶段安排

### 第一阶段：先跑通每周自动周报

主要工作：

- 新增 GitHub Actions 每周任务。
- 配置 API Key 和飞书凭证。
- 新增飞书发布脚本。
- 建立飞书“每周简报索引”表。
- 跑通一次完整流程。

这一阶段的目标不是做复杂系统，而是先替代 Colab 和人工上传。

### 第二阶段：补齐通知和异常处理

主要工作：

- 成功后自动发飞书群通知。
- 失败后记录错误信息，必要时提醒负责人。
- 保留每周产物和质检文件。
- 持续校准强相关和中等相关判断。

### 第三阶段：评估是否迁移服务器

如果后续出现以下情况，再考虑轻量云服务器：

- GitHub Actions 经常超时或失败。
- 每次运行超过 40-60 分钟。
- 需要长期保存数据库和完整日志。
- 后续要增加更多来源、全文抓取或内部检索网页。

---

## 九、结论

1. 正式运行频率改为每周一次，不再做每日定时抓取。
2. 第一阶段先用 GitHub Actions 自动运行，并把结果发布到飞书。
3. 员工阅读入口放在飞书，GitHub 只作为后台运行环境。
4. Colab 保留为调试工具，不再承担生产任务。
5. 等每周流程稳定后，再根据运行时长和维护需求决定是否迁移到云服务器。
