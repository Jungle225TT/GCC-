## 1. 项目背景与核心问题

### 我们在解决什么问题

成都创新金融研究院的研究人员需要持续跟踪 GCC（海湾合作委员会）地区的政策研究动态。这个地区的主要智库产出以**英文**为主，分布在沙特、阿联酋、卡塔尔、科威特、巴林、阿曼六个国家，加上黎巴嫩、埃及等泛 MENA 国家，以及美国、英国的域外视角智库。

手动跟踪的痛点：
- 来源分散，29个智库各有独立网站，每日手动查阅不现实
- 语言门槛，英文研究文章需要初步筛选才能分配阅读
- 信噪比低，很多智库网站混杂了活动公告、播客、人物介绍等非研究内容
- 重复劳动，同一篇文章可能被两个人同时标记

### 系统的核心设计目标

1. **自动化覆盖**：每日无需人工干预，自动抓取并筛选
2. **高相关性**：尽可能只保留实质性研究文章，过滤噪音
3. **可维护性**：筛选规则应该可以快速调整，不需要每次改 Python 代码
4. **低成本**：AI 调用成本控制在每日 $0.005 以内
5. **团队可用**：无需本地 Python 环境，研究员通过浏览器即可操作

## 2. 整体架构设计

### 单文件主脚本的选择

整个系统的核心逻辑全部在 `gcc_thinktank_scraper_v2.py` 一个文件中。这是有意为之，而非偷懒。

**理由：**
- 系统的主要使用场景是 Colab 笔记本，每次运行环境是临时的，单文件比多模块更容易管理依赖
- 调试时不需要跨文件追踪调用链
- 研究员或维护者克隆仓库后，不需要理解模块结构就能找到关键参数

唯一例外是 `ai_client.py`，它被独立出来是因为 DeepSeek / Anthropic 的切换逻辑需要被 `feishu_sync.py` 和主脚本共同使用，避免重复代码。

### 数据流

```
29个智库网站
     |
     v
[抓取层] RSS 优先，HTML 兜底，Playwright 处理 SPA
     |
     v
[漏斗层] 四层筛选（来源可信度 -> 关键词 -> 内容类型 -> AI）
     |
     v
[后处理] 跨来源去重 -> AI筛选 -> SQLite增量去重 -> 无日期过滤 -> 时效过滤
     |
     v
[导出层] Markdown报告 + JSON数据 + AI简报MD + AI简报PDF
     |
     v
[分发层] 飞书多维表格（可选）
```

## 3. 智库覆盖策略

### 为什么是 29 个，不是更多

**智库的纳入标准：**
1. 有英文出版物（中文/阿拉伯文暂不纳入）
2. 发布频率较高（月均至少 2 篇实质性研究文章）
3. 有可稳定抓取的公开页面（无需登录）
4. 研究议题与 GCC 地区有直接或间接关联

扩展到更多智库会增加噪音和维护成本，目前 29 个覆盖了 GCC 研究领域最重要的来源。

### core_gcc 与 pan_mena 的分层逻辑

系统将智库分为两个 tier，决定了漏斗的进入路径：

**core_gcc（17个）**：GCC 六国本地智库 + 专注 GCC/海湾地区的域外机构（如 AGSIW）。这些机构发布的内容天然与 GCC 高度相关，因此**直接赋分 99，绕过关键词评分**。对它们的过滤只能通过"硬过滤"（第三层）进行，而不能用降权词——因为降权只能将 99 降到 97，远超通过阈值 3，毫无效果。

**pan_mena（12个）**：覆盖整个中东/北非的广谱智库（如 Carnegie MEC、MEI、CSIS）。这些机构会发布大量与 GCC 无关的内容（叙利亚、利比亚、摩洛哥等），因此需要通过关键词评分过滤，只保留评分 ≥ 3 的文章。

这个分层是最重要的架构决策之一：**它决定了过滤噪音的方式必须根据 tier 来选择。**

### deep_topic 专题页机制

Carnegie MEC、Al Sharq Forum、Arab Reform Initiative 这三个泛 MENA 智库比较特殊：它们有按国家/地区组织的专题页（如 Carnegie 的 `/region/gulf-states/`），直接从这些专题页抓取，等效于 core_gcc 的效果——内容已经过人工分类，不需要关键词评分。

这是一个精准度与便利性的平衡：不是所有 pan_mena 智库都有这样的专题页，能用时优先用。

## 4. 抓取层设计：RSS + HTML 双通道

### 为什么优先 RSS

RSS 的优势是确定性：每条条目都是一篇文章，标题、URL、日期字段是结构化的，不需要猜测 CSS 选择器，也不会抓到导航链接。

实践中发现，GCC 地区智库的 RSS 质量差异很大：
- KAPSARC、Derasat 的 RSS 质量高，摘要完整
- 部分智库的 RSS 只有标题，没有摘要（影响关键词评分）
- 部分智库的 RSS 已停止维护或返回错误格式

系统的策略是：**RSS 有效（返回 ≥ 3 条有效条目）则跳过 HTML 抓取**，否则回退到 HTML。这个阈值（3条）是为了过滤掉"RSS 存在但实际上只有 1 条占位内容"的情况。

### HTML 抓取的难点

HTML 抓取的核心挑战是区分"真正的文章链接"和"导航/介绍链接"。一个智库首页通常包含：
- 最新文章链接（目标）
- 栏目导航（Publications、Events、About Us）
- 标签/分类页链接（Energy, Security）
- 作者页、机构介绍页

`_is_likely_article()` 函数负责这个判断，依据是：
1. 标题黑名单（`filter_rules.yaml` 的 `nav_exact`）
2. URL 路径模式（`filter_rules.yaml` 的 `nav_url_patterns`）
3. 标题长度（过短的标题通常是导航词）
4. 日期字段是否能解析（导航页通常没有日期）

这些规则都提取到 `filter_rules.yaml`，运营人员无需改代码即可维护。

### Playwright 的使用边界

Playwright（Chromium headless）用于处理 JavaScript 渲染的 SPA 网站。GCC 地区有几个智库（Future Center、EPC 的部分页面）使用了前端框架渲染内容，普通 requests 只能拿到空 HTML。

但 Playwright 有明显缺点：
- 启动慢（每次约 2–5秒）
- Colab 免费版内存有限，大量并发会崩溃
- 不是所有 Colab 环境都能顺利安装 chromium

因此 Playwright 是**可选的、按需启用的**，不是默认开启。系统对每个智库单独标记 `"use_playwright": True`，只有这些智库才会用到浏览器，其余仍用 requests。

## 5. 四层漏斗：为什么这样设计

漏斗的每一层针对不同类型的噪音，顺序设计也有讲究。

### 第一层：来源可信度（为什么放最前）

来源可信度层最简单，几乎不消耗计算资源，放在最前面可以**让 core_gcc 的文章直接跳过后面的评分层**，节省关键词匹配的计算量，更重要的是避免因关键词覆盖不全而误判 core_gcc 的文章。

例如，一篇关于巴林能源政策的文章标题可能是 "BAPCO Refinery Expansion: Strategic Implications"，完全没有 GCC、海合会等关键词，关键词评分会是 0 分——但这显然是核心内容。来源可信度层解决了这个问题。

### 第二层：关键词评分（为什么是 3 档权重）

关键词分为三个权重档：

| 档位 | 权重 | 典型词 | 设计理由 |
|------|------|--------|---------|
| STRONG | 3分 | gcc、gulf cooperation council、hormuz | 命中即高度相关，一个词就接近阈值 |
| COUNTRY | 2分 | saudi arabia、uae、qatar | 命中说明在谈 GCC 国家，但可能是边缘话题 |
| WEAK | 1分 | gulf、middle east、mena | 泛指，需要多个才能通过 |

标题命中翻倍（×2）是因为标题通常是作者对文章核心议题的凝练，标题里出现 GCC 比摘要里出现更有价值。

阈值 3 是经过 `scoring_test.py` 在 38 篇人工标注样本上测试得出的最优值，在此阈值下精度达到 100%（0 篇不相关文章漏过），召回率略有牺牲（7/10 边界文章通过）。

### 第三层：内容类型（为什么是关键词匹配而非 AI）

第三层判断文章是否属于"应该被丢弃的内容类型"（播客公告、活动公告、人物简介等），用的是词表匹配而非 AI。

这个选择基于两个判断：
1. **可预测性**：这类噪音有固定的语言模式（"Episode X of"、"organizes a roundtable"、"Dr. X is a Fellow"），正则/词表匹配准确率很高
2. **成本**：如果用 AI 来判断，每篇文章都要调用一次 API，成本增加 10 倍

第三层的另一个关键设计是：**它对所有 tier 生效，包括 core_gcc**。这是唯一能过滤 core_gcc 噪音的机制。当某个 GCC 本地智库开始频繁发布播客公告时，把关键词加入 `hard_filter_words` 即可，不需要其他改动。

### 第四层：AI 辅助（为什么是可选的）

AI 层处理的是"评分在阈值附近的边界文章"——这些文章关键词评分勉强通过，但可能实际上与 GCC 关联很弱（比如一篇谈伊朗核问题但只是顺带提到海湾紧张局势的文章）。

AI 层是可选的原因：
- 不是每次运行都需要 API 调用
- 对于 core_gcc 智库，这层没有意义（它们已经在第一层通过了）
- 本地调参/测试时不想消耗 API 额度

实践中，AI 层的主要价值在于提升 pan_mena 智库的精度，减少边界文章的误收录。

## 6. 关键词体系：配置化取代硬编码

### 为什么引入 keywords.yaml

v2.4.1 之前，过滤词和降权词是硬编码在 Python 文件里的列表。每次需要加一个过滤词，都要改代码、重新运行、观察效果，反馈周期慢。

`keywords.yaml` 的引入让这个过程变成：
1. 编辑 YAML 文件
2. 运行 `--dry-run-keywords` 查看效果
3. 确认后正式运行

不需要懂 Python，研究员也可以参与词表维护。

### 三种词表的设计取舍

**hard_filter_words（硬过滤）**
命中即丢弃，无视来源 tier。适合那些"无论来自哪个智库都不想要"的内容类型，如播客、活动公告。这是唯一对 core_gcc 有效的过滤手段。

**demote_words（降权，检查标题+摘要）**
命中则评分 -2，不是直接丢弃。适合"通常不相关但偶尔相关"的词，比如 "Western Sahara"（大多数时候不相关，但偶尔有文章讨论摩洛哥/海湾关系时会顺带提到）。直接丢弃会有误伤风险，降权是更保守的选择。

**title_only_demote_words（仅标题降权）**
这是 v1.3 新增的，因为遇到了一个具体问题：NATO 这个词被放入 `demote_words` 后，检查标题+摘要，导致大量谈海湾安全的文章因为摘要里引用了"NATO 框架"、"NATO 式安全保障"等表述而被降权。但如果标题是"NATO's Gulf Strategy"，确实应该降权（这篇文章的主角是 NATO 而非 GCC）。

`title_only_demote_words` 让我们可以精细控制：**只有当这个词出现在文章标题时才降权**，摘要里的引用不触发。

### dry-run 模式的设计理由

`--dry-run-keywords` 模式会输出两个文件：
- `keyword_dryrun.json`：每篇通过的文章及其降权明细（哪些词触发了降权、降了多少分）
- `filtered_out.csv`：被硬过滤拦截的文章及触发原因

这两个文件是词表调参的核心工具。没有这个模式，调参只能靠"运行 → 查结果 → 猜测"，效率极低。

## 7. AI 层：DeepSeek 而非 Claude

### 为什么默认使用 DeepSeek

系统支持两个 AI provider：DeepSeek（默认）和 Anthropic Claude（可选）。默认选择 DeepSeek 的理由：

**成本**：DeepSeek 的 API 成本约为 Claude Sonnet 的 1/10 到 1/20。对于每日批量处理 50–200 篇文章的场景，这个差距非常显著。每日运行成本 < $0.005 是一个关键约束，决定了系统能否长期可持续运行。

**性能**：对于边界文章分类和标题翻译这类任务，DeepSeek 的质量与 Claude 接近，不存在明显差距。

**API 兼容性**：DeepSeek 使用 OpenAI 兼容的 API 格式，`ai_client.py` 可以用 `openai` 库对接，无需额外 SDK。

### 什么时候应该切换到 Claude

Claude 的优势在复杂推理任务上。以下情况可以考虑切换：

- 生成 AI 简报时，如果发现 DeepSeek 对某些政策背景的理解不够深入
- 需要更准确的中文翻译（Claude 的中英互译质量略好）
- 批量处理量较小、对成本不敏感时

切换方式：
```bash
export AI_PROVIDER=anthropic
export ANTHROPIC_API_KEY="sk-ant-xxxxx"
```

### AI 简报为什么分批处理

`generate_ai_summary()` 将文章分为每批 10 篇分批调用 AI。原因：

- 一次性发送 50 篇文章的内容会超过大多数模型的 context window 限制（或触发超长 prompt 的额外费用）
- 每批 10 篇约消耗 3000–6000 tokens，每篇文章有充足的 token 空间做详细分析
- 分批处理也便于在某批失败时只重试该批，不影响其他批次

## 8. 相关性两档分级：从三档到两档的演变

### 早期的三档设计

系统早期将文章分为三档：
- 优先（Priority Read）
- 推荐（Recommended）
- 备查（Reference）

这个设计看起来更精细，但实际使用中发现几个问题：

1. **中间档无明确行动指导**："推荐"到底该不该读？三档的区分对读者的实际行为影响不大
2. **维护复杂**：三个阈值需要三个判断条件，每次调参要同时考虑三个边界
3. **报告结构复杂**：Markdown 报告需要三个区块，视觉层次反而不清晰

### 两档的设计逻辑

简化为两档后，规则变得非常清晰：

- `topic_relevance_score >= 4.0` → **强相关**（必读）
- `topic_relevance_score < 4.0` → **中等相关**（选读）

这里用的是 `topic_relevance_score`（0–5 综合评分），而不是漏斗第二层的 `keyword_score`。两者的区别：
- `keyword_score`：纯关键词匹配的原始分，是机械性的
- `topic_relevance_score`：结合来源类型（core_gcc / pan_mena）、内容类型（report/analysis vs newsletter）等因素的综合分，更能反映文章实际的研究价值

中等相关保留而不丢弃，是因为它们仍然是合格的 GCC 研究文章，只是核心程度略低。用户可以根据时间决定是否阅读第二档。

### PDF 中的视觉呈现

Markdown 文件里用 `⭐` 和 `📄` 区分两档，在 PDF 渲染时遇到了字体问题：reportlab 使用的 STSong-Light 是 CID 字体，不包含 emoji 字符集，这两个符号会显示为乱码或空白。

解决方案是在 PDF 渲染层（`export_summary_pdf`）增加一个 `_colored_symbols()` 函数，在 XML 标记组装阶段将这两个 emoji 替换为带颜色的实心圆（reportlab 支持 `<font color="...">` 标签）：
- `⭐` → 红色 `●`（表示强相关，视觉上"重要/警示"）
- `📄` → 蓝色 `●`（表示中等相关，视觉上"普通/信息"）

这个替换在 `_safe()`（处理普通文本）和 `_with_links()`（处理含链接的文本）两条路径中都有应用，确保所有出现这些符号的地方都能正确渲染。

## 9. AI 简报设计：为什么不只是摘要

### 结构化分析 vs 摘要翻译

早期版本的 AI 功能只是翻译标题、生成一句话摘要。这对研究员的价值有限：他们真正需要的是"这篇文章对我的研究有什么意义"，而不是"这篇文章说了什么"。

AI 简报的每篇文章固定输出五个维度：

| 维度 | 说明 |
|------|------|
| 核心议题 | 作者的主要研究问题 |
| 主要判断 | 作者的核心论点与论证逻辑 |
| 对GCC地区的影响 | 从经济、政治、安全等维度分析影响路径 |
| 对华关联 | 与中海/中阿关系、能源合作、一带一路的关联 |
| 关键数据或事件 | 文章中出现的具体数字、时间节点 |

"对华关联"这个维度是专门为研究院的工作目标定制的——研究院的核心任务是分析 GCC 地区动态对中国的影响，这个维度直接对接研究员的实际需求。

### 趋势信号作为第三步

简报的第三步（趋势信号）独立于逐篇解析，在所有文章分析完成后额外调用一次 AI，要求它跨文章归纳 3–5 条议题趋势。

这个设计的价值是：**跨文章的共性信号往往比单篇文章更有价值**。如果本期有 5 篇文章都在谈海湾安全架构重组，但每篇角度不同，趋势信号层可以将这个共性显式化。

## 10. PDF 生成：技术选择与已知问题

### 为什么用 reportlab 而非其他方案

备选方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| reportlab | 无需外部程序，纯 Python，Colab 可用 | 不支持完整 Markdown，需自己实现渲染 |
| WeasyPrint | HTML/CSS 驱动，排版精确 | 需要 libcairo 等系统库，Colab 安装困难 |
| pandoc | 功能强大，支持多种格式 | 需要系统级安装，Colab 环境麻烦 |
| fpdf2 | 比 reportlab 轻量 | 中文支持差 |

reportlab 的核心优势是 **CID 字体支持**（STSong-Light 宋体）——这是在无需安装额外字体文件的情况下，Colab 环境里唯一能可靠渲染中文的方案。

### STSong-Light 的局限

STSong-Light 是一个老式宋体 CID 字体，包含基本汉字，但不支持：
- Emoji（⭐、📄等）
- 部分生僻汉字
- 阿拉伯文（部分智库名称含阿拉伯文）

因此凡是在 PDF 中出现 emoji 的地方，都需要在渲染前替换为等价的纯文本表示（见第8节）。

### 降级模式

PDF 生成有两层保护：
1. 首次生成失败时，自动降级为"纯文本链接模式"重试（去掉所有 `<a href>` 标签，保留链接文字）
2. 降级仍失败，返回空字符串，主流程继续运行，不中断整体输出

这个设计是因为 PDF 是"锦上添花"的功能——Markdown 简报已经可以满足阅读需求，PDF 损坏不应该导致整次运行白费。

## 11. 去重机制：两层防护

### 为什么需要两层去重

**第一层（跨来源 in-memory 去重）**：在一次运行内，同一篇文章可能被两个智库各自抓到（例如一篇关于 GCC 峰会的文章同时出现在卡塔尔智库和一个域外美国智库的推荐列表中）。这层去重在 `run_scraper()` 抓取结束后、AI 筛选前进行，用 URL 做唯一标识，始终生效（即使 `--no-dedup` 也不跳过）。

**第二层（SQLite 增量去重）**：跨运行去重，防止今天已推送到飞书的文章明天再次出现。这层用 `gcc_dedup.db` 存储已见 URL，有时间窗口（默认 1 天），`--no-dedup` 可以跳过。

两层去重的关系：第一层是运行内的防重，第二层是跨运行的防重，各司其职。

### SQLite 而非其他存储

去重数据库选择 SQLite 的理由：
- 无需额外服务，Python 内置支持
- 文件型，可以直接备份或删除重置
- Colab 环境与 Google Drive 挂载后，数据库文件可以持久化到 Drive

## 12. Colab 运行环境：为什么不用本地服务器

### 当前团队的实际情况

研究院的研究员不是技术人员，没有维护 Linux 服务器的能力，也没有统一的 Python 开发环境。购买云服务器则增加了运维负担。

Colab 免费版提供了：
- Python 环境 + 主流包预装
- 可挂载 Google Drive（持久化数据）
- 浏览器访问，无需本地配置
- 共享笔记本，团队协作方便

代价是：
- 每次连接需要重新安装部分依赖（约 2–3 分钟）
- 不支持真正的定时任务（需要手动运行或配合 GitHub Actions）
- 免费版资源有限，大规模 Playwright 并发容易超内存

### @param 控件 vs ipywidgets

Colab 笔记本里的参数控件使用 `# @param` 注释风格，而不是 ipywidgets。原因：

`@param` 是 Colab 原生支持的特性，无需安装额外包，渲染速度快，在 Colab 界面里有更好的原生集成（下拉框、复选框等）。ipywidgets 在 Colab 里偶尔有显示问题，且需要额外安装。

### Colab Secrets 管理 API Key

Colab 提供了 Secrets 功能（左侧 🔑 图标），API Key 存在 Secret 里而不是直接写在笔记本代码里，好处是：
- 笔记本可以安全分享给团队成员，不会泄露 Key
- 不同成员可以配置自己的 Key
- 不会因为笔记本被 commit 到 GitHub 而意外泄露

## 13. 已知局限与未来方向

### 当前局限

**抓取层**
- 部分网站有反爬机制（HTTP 403），目前无法处理，只能定期人工检查是否失效
- Playwright 在 Colab 免费版的内存限制下，不能并发运行多个智库，目前是串行的
- RSS 摘要有时被截断，影响关键词评分

**筛选层**
- 关键词评分无法理解上下文语义（"Gulf of Mexico"和"Arabian Gulf"都包含"Gulf"）
- AI 层的边界判断依赖模型的背景知识，对新出现的地缘政治事件可能反应迟钝

**输出层**
- PDF 的目录链接在 Colab 环境预览时不可点击（是 PDF 内部锚点，需要 PDF 阅读器支持）
- STSong-Light 的排版质量有限，不适合正式出版物

### 值得探索的方向

**短期**
- 对评分分布做可视化（每次运行后输出强/中分布直方图），帮助判断词表是否需要调整
- 飞书同步增加"强相关"字段，方便在飞书表格里直接筛选

**中期**
- 引入向量嵌入（embedding）做语义相关性计算，替代或补充现有的关键词评分，提升泛 MENA 智库的召回精度
- 对 AI 简报的质量做反馈收集，用于改进提示词

**长期**
- 如果研究院的服务器资源允许，可以迁移到定时任务（cron + GitHub Actions），实现真正的无人值守日报
- 阿拉伯文智库的覆盖（目前只抓英文，海湾本地官方智库有部分内容只有阿拉伯文版）
